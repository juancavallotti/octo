Guide

Authoring with AI assistants

Octo ships a Model Context Protocol (MCP) server. Point any MCP-capable assistant — Claude, an IDE agent, your own client — at the /mcp endpoint and it can read the runtime's block catalogue, study worked examples, draft and validate a definition, create the integration, run it in a dev runner, read the logs, and iterate — all without leaving the conversation.

The idea

A grounded authoring loop.

An assistant guesses badly when it has to invent a YAML schema from memory. The MCP server hands it the ground truth — the live capability catalogue and real examples — and a validate tool to check its work, so it converges on a definition that actually builds.

📚

Grounded

The exact blocks, connectors, and fields the runtime supports — served as a resource the model reads before authoring.

✅

Validated

A draft is checked against the runtime schema before it's ever saved. Validation is advisory — it guides, it doesn't gate a run.

▶️

Runnable

The assistant starts the integration in a per-session dev runner, gets a test URL back, and tails the logs to debug.

What the server exposes

Tools.

Two families: authoring tools that read and write integration definitions, and run-control tools that drive a dev runner scoped to the client's session.

Authoring

list_integrationsList saved integrations as {id, name}.
open_integrationRead one by id → {id, name, definition} (the runtime YAML).
validate_definitionCheck a draft YAML against the runtime schema without saving; returns descriptive errors.
create_integrationCreate from a name + definition. Returns the created record.
update_integrationOverwrite a definition (and optionally rename).

Run control (per session)

can_start_integrationIs a runner available and does the definition validate? → {available, valid, errors}.
run_integrationStart (or restart) the integration in the dev runner, optionally injecting env vars. Returns a test URL.
get_run_logsRead this session's runner log buffer, oldest line first.
stop_integrationStop this session's run and free its resources.
Session isolation. Each client gets its own runner and log buffer, keyed by the MCP session. Two assistants connected at once won't trample each other's runs. A test URL is only returned when the integration declares an HTTP_PORT (i.e. it's a network service); otherwise the run is internal-only.
Context the model reads

Resources & prompts.

Beyond tools, the server publishes resources the assistant pulls into context, and a guided prompt that walks it through the whole loop.

Resources

URIWhat
octo://runtime/schemaThe catalogue of blocks & connectors with their configurable fields. Read this first.
octo://examplesIndex of worked examples — each lists the blocks it shows and its read URI.
octo://examples/<slug>A complete example definition. Slugs: hello-world, builtins, http-orders, error-handling, ai-router.

Prompt

create-integration — a step-by-step prompt (with an optional goal argument) that tells the assistant how to do this well: read the schema, study an example, draft and validate_definition, create or update, then run_integration and iterate from get_run_logs. Selecting it primes the model with the right sequence.

Connecting

The /mcp endpoint.

Both the hosted platform and the standalone app expose the server at /mcp over the streamable-HTTP transport. The difference is authentication.

🌐

Platform — API-key gated

The hosted app authenticates each request with a per-user API key as a bearer token. The route owns its own auth and bypasses the OIDC session gate; an invalid or missing key gets a 401 with a bearer challenge.

https://<your-platform-host>/mcp
Authorization: Bearer octo_xxxxxxxx…
💻

Standalone — local & open

The standalone app serves /mcp unauthenticated for local use — point your client at http://localhost:3000/mcp and go. No keys, no accounts; ideal for a desktop authoring loop.

http://localhost:3000/mcp
# no auth header needed
On the platform

Users & API keys.

The platform bootstraps a user account on first sign-in. From there you mint the keys that authenticate MCP clients.

Managing keys

The Account → API keys page lists your active keys (name, prefix, last-4, created/expiry, last-used) and lets you create and revoke them. A new key's full token is shown once at creation and never again — copy it into your MCP client then.

How a key is checked

On each request the platform verifies the bearer token against the orchestrator, which resolves it to its owner before the request reaches the MCP layer. Keys can carry an expiry; revoking one takes effect immediately.

Wiring. The platform MCP route needs ORCHESTRATOR_URL (for integration CRUD and key verification) and OCTO_BIN_PATH (so the run-control tools have an octo binary to launch). Set OCTO_DOCS_URL to have the authoring prompt link back to these docs.
