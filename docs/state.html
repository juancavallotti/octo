Guide

State & clustering

Flows are stateless by default — but real integrations need to remember a cursor, cache an expensive call, hold a token, or make sure a scheduled job fires once even when the integration runs as several replicas. Octo's runtime services give blocks a key–value store, an encrypted secret store, and leader election, behind one interface that works the same in a single process and across a cluster.

What you get

Three primitives.

Runtime services are injected into every flow's execution context. Blocks reach them through stable interfaces, so a flow doesn't know — or care — whether it's running on your laptop or in Kubernetes.

🗃️

Key–value store

Versioned Get / Set / Delete with optimistic concurrency. The object-* and cache blocks build on it; you rarely touch it directly.

🔐

Secret store

The same KV API, but routed to encrypted namespaces. The http-client OAuth2 token cache lives here; so can anything a connector must keep private.

👑

Leader election

Acquire a key and ask IsLeader(). Exactly one replica wins per key — the basis for distributed cron and any run-once work.

The store

Key–value, namespaced & versioned.

Keys live in namespaces that isolate concerns. Your flows touch the user space through the store blocks; the runtime keeps its own state in system; secrets ride in encrypted *_secrets twins of each.

Namespaces

NamespaceFor
userYour flow state — the object-read/write/delete and cache blocks.
systemInternal runtime & connector state. User blocks must not read it.

How writes stay consistent

Every entry carries a version. A write passes the version it expects: 0 creates, a positive value updates only if it still matches. A mismatch returns a version conflict, so two replicas racing on the same key can't silently clobber each other — object-write retries the read-modify-write for you.

The store has no native TTL. The cache-scope block layers expiry on top by stamping a deadline into the value it stores, and treating expired entries as a miss.

Block reference. The object-read, object-write, object-delete, cache-scope, and invalidate-cache blocks — with their settings — are documented under State & cache blocks.
Encrypted at rest

The secret store.

The secret store is the same KV interface with one twist: it transparently rewrites a namespace to its *_secrets counterpart and encrypts values before they're persisted. Plain KV traffic pays no encryption cost; secrets are never written in the clear.

🔑

The encryption key

On the platform the orchestrator owns the cipher. It reads a base64 KV_ENCRYPTION_KEY (16/24/32 bytes → AES-128/192/256-GCM) and encrypts every value bound for a secret namespace. The deploy tooling generates and wires this key for you. If it's absent, secret namespaces are disabled and writes to them fail loudly.

♻️

Who uses it

The http-client OAuth 2.0 auth mode caches its access/refresh tokens here, keyed per connector, so tokens survive restarts without ever appearing in logs or config. Connectors that must persist credentials use the same path.

Run exactly once

Leader election & distributed cron.

Scale an integration to three replicas and a naive cron source would fire three times. Octo prevents that with leader election: the cron source campaigns for a per-schedule key and only the leader emits a tick — the others skip it.

The pattern

Acquire a Leadership handle for a key once, then gate exclusive work on a cheap, cached IsLeader() check. The cron source does exactly this — keyed by its connector name — so the schedule fires once cluster-wide regardless of replica count.

Under the hood

In the Kubernetes provider, leadership is backed by a coordination/v1 Lease per key, scoped to the deployment (15s lease, 10s renew, 2s retry). In standalone mode a single process is always the leader — same code, no coordination needed.

Same interface, two backends

Standalone vs Kubernetes.

The active provider is chosen at startup by RUNTIME_SERVICES_MODULE (default standalone). The CLI builds services once and reuses them across hot-reloads, so leases and connections aren't churned.

Wiring (Kubernetes)

The orchestrator injects these into each deployed pod when runtime services are enabled (set its ORCHESTRATOR_URL to turn the feature on; the module then defaults to k8s):

RUNTIME_SERVICES_MODULEk8s
OCTO_DEPLOYMENT_IDScopes every KV key and Lease to this deployment.
ORCHESTRATOR_URLThe KV API endpoint the provider calls.
POD_NAME / POD_NAMESPACEPod identity (downward API), used by leader election.
It degrades gracefully. If no provider is wired, blocks fall back to a no-op store (writes vanish) and always-leader election. That keeps a flow runnable anywhere; just don't rely on persistence until a real provider is in place.
Try it

The runtime-services sample.

samples/runtime-services.yaml ties it together: a 5-second ticker reads a counter from the store, bumps it, writes it back, then produces a report inside a cache-scope so the body is computed once and served from cache until its 1m TTL lapses. A reset flow invalidates the cache; a fetch flow calls an OAuth2-protected API through the secret-store-backed token cache.

# Run it (standalone services, in-process store)
task run:sample -- runtime-services.yaml

# Bust the cache so the next tick recomputes
./bin/octo invoke --config samples/runtime-services.yaml --flow reset
